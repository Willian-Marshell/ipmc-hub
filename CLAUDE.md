# IPMC Hub — Claude 协作指南

## 项目背景

深圳大学"智能感知与运动控制（IPMC）"本科微专业的学生开源知识库。
目标是把课程经验、比赛复盘、AI 工具心得、升学攻略沉淀下来，代代接力。
受众是 IPMC 在读及候选学生，以及对该方向感兴趣的非微专业学生。

项目愿景：万事开头难，但只要把规范化守则和激励机制立起来，每届都能在前人基础上继续叠加，形成可持续的知识库。

## 技术栈

- Docusaurus 3.10.1 + TypeScript + React 19
- 包管理：npm（不要用 yarn / pnpm）
- Node 要求：>= 20
- 部署：GitHub Actions → `gh-pages` 分支，触发条件 push to `main`
- 仓库：`Willian-Marshell/ipmc-hub`（注：仓库会从 `my-blog` 改名为 `ipmc-hub`）
- 部署 URL：`https://willian-marshell.github.io/ipmc-hub/`
- 站点语言：仅 `zh-Hans`，不启用 i18n

## 常用命令

- `npm run start` 本地开发（默认 3000 端口）
- `npm run build` 生产构建
- `npm run typecheck` TypeScript 类型检查
- `npm run serve` 预览构建产物
- `npm run clear` 清理 Docusaurus 缓存

## 内容归属判定（重要）

每次新建内容前先判定归属：

- 教材式 / 长期演进 / SOP / 规范 → `docs/`
- 个人复盘 / 时间线 / 一人一文一署名 → `blog/`
- 拿不准时反问用户，不要擅自归类

举例：
- 「嵌入式课程通关笔记」→ `docs/courses/embedded-systems/`
- 「我大三准备保研的半年复盘」→ `blog/`
- 「RoboMaster 视觉组技术栈」→ `docs/competitions/robomaster/`
- 「我们队拿了二等奖的赛后感想」→ `blog/`

## 目录结构约定

```
docs/
├── intro.md                      项目总览（落地页）
├── courses/<course-slug>/        课程笔记按课程分
├── competitions/<event-slug>/    比赛三件套：getting-started / tech-stack / pitfalls
├── ai-toolkit/                   AI 工具与 Agent 心得
├── postgrad/                     保研 / 考研 / 留学 / 就业
├── contributing/                 贡献规范（六篇必读）
└── archive/<year>/               退役内容（只搬不删）
```

- 图片放 `static/img/<doc-slug>/`，引用用站点绝对路径 `/img/<doc-slug>/<file>`
- 禁止外链图床（链接会失效，违背"代代接力"原则）；禁止 base64 内嵌（污染 markdown 源文件）
- 多图导入用 `npm run add-images -- --doc <文档路径> <图片...>`，脚本会自动建目录、改名、输出可粘贴的 Markdown 引用（详见 [docs/contributing/02-writing-style.md](docs/contributing/02-writing-style.md) 图片小节）
- 例外：贡献者通过 GitHub 网页编辑器拖图产生的 `user-images.githubusercontent.com` 链接是临时合法的，但合入后栏目主编需用 `add-images` 脚本迁移到仓库；遇到这类 PR 请提醒迁移
- 修改任何贡献规则必须同步更新 `docs/contributing/`

## frontmatter 硬规则

新建文档必须带完整 frontmatter，缺一不可：

```yaml
---
title: 嵌入式系统课程通关笔记
description: 一句话摘要，会出现在搜索结果与卡片
authors: [zhangsan]            # 必须先在 blog/authors.yml 注册
cohort: 2024                   # 作者届数（入学年份）
course: embedded-systems       # 关联课程 slug，可选
tags: [课程心得, 嵌入式]
status: stable                 # draft | stable | outdated | archived
last_reviewed: 2026-05-30      # 退役机制核心字段
reviewers: [lisi]              # 终审人 GitHub ID，可选
sidebar_position: 2            # 仅 docs 用
---
```

字段说明：
- `authors` 引用 `blog/authors.yml` 里的 key，作者首次贡献必须先在 authors.yml 自助登记
- `cohort` 写入学年份，例如 2024 级写 `2024`
- `status` 流转：`draft → stable → outdated → archived`
- `last_reviewed` 为最近审核日期，超 18 个月会被自动标记为 `outdated`

## 届数处理

统一用 frontmatter `cohort: <year>` + 标签 `grade-<year>` 双标，
**不**按届建目录（避免空壳累积）。

## 标签体系（blog/tags.yml）

三轴正交：
- 类型轴：`课程心得` `比赛复盘` `升学经验` `AI实践` `生活感悟`
- 届数轴：`grade-2024` `grade-2025` …
- 主题轴：`嵌入式` `RoboMaster` `保研` `考研` `留学` …

## sidebar 维护

`sidebars.ts` 拆成两个具名 sidebar：
- `knowledgeSidebar`：知识内容（intro + 课程/比赛/AI/升学/归档），用 category 节点包裹 autogenerated，`navbar.docSidebar` 绑到这里
- `contributingSidebar`：贡献指南，整个 `contributing/` 目录 autogenerated

调顺序 / 改标题统一通过：
- 各目录下的 `_category_.json`（控制目录显示名、顺序、折叠）
- 文档 frontmatter 的 `sidebar_position` / `sidebar_label`

**新增顶层目录**（如新加一个 `docs/research/`）时，需要在 `sidebars.ts` 的 `knowledgeSidebar` 里添加对应 category 节点 + `_category_.json`，并确保 `link.slug` 与 footer/首页里的 `/docs/category/xxx` 链接对得上。子目录与文档新增完全自动化，不需要改 sidebar。

## 配置安全区

未经用户明确要求，**不修改** `docusaurus.config.ts` 中的：

- `url`
- `baseUrl`
- `organizationName`
- `projectName`
- `i18n`

这些字段与 GitHub Pages 部署强耦合，改错会直接 404。

## 退役策略

内容只搬不删：

1. 过时但仍有参考价值：`status` 改为 `outdated`，加页头提示横幅
2. 完全失效：移入 `docs/archive/<year>/`，保留原文件名，`status` 改为 `archived`
3. **任何"删除文档"的请求都要先反问能否归档**，删除是最后手段

归档内容仍保留在 sidebar（折叠分组）和搜索结果中，便于历史追溯。

## 文件名与 URL 的关系（重要）

Docusaurus 默认会**剥掉文件名的数字前缀**作为 URL slug。
例如 `docs/contributing/01-quickstart.md` 的实际 URL 是 `/docs/contributing/quickstart`，**不是** `/docs/contributing/01-quickstart`。

写内部链接时一律用**剥掉前缀的形式**，否则会触发 build 时的 broken link 检查（`onBrokenLinks: throw`）。

## 代代接力机制

- `docs/contributing/` 六篇必读（文件名带数字前缀控制侧栏顺序，URL 自动去前缀）：
  1. `01-quickstart.md` → `/docs/contributing/quickstart`
  2. `02-writing-style.md` → `/docs/contributing/writing-style`
  3. `03-frontmatter-spec.md` → `/docs/contributing/frontmatter-spec`
  4. `04-review-flow.md` → `/docs/contributing/review-flow`
  5. `05-archival-policy.md` → `/docs/contributing/archival-policy`
  6. `06-star-program.md` → `/docs/contributing/star-program`
- 三级 review：作者自查 → 同届互评 → 栏目主编（CODEOWNERS）终审
- `.github/PULL_REQUEST_TEMPLATE.md` 提供贡献勾选清单（待补）

## 写作风格

- 标题用动词或问句，避免名词堆砌
- 代码块必须标语言：` ```python ` / ` ```bash ` / ` ```ts `
- 外部引用注来源链接
- 转述学长经验必署名（写明届数与 GitHub ID）
- 不用浮夸营销话术，不堆黑话
- 涉及具体课程 / 老师 / 课程内容时务必客观，避免主观贬低
- blog 摘要分隔符用 `{/* truncate */}` 而**不是** `<!-- truncate -->`（项目启用了 MDX，HTML 注释会让 build 报 `Unexpected character` 错）

## 验证流程

- 改完任何 `.md` / `.mdx` / `.tsx` / config 后必须跑 `npm run build`，build 失败的改动不算完成
- 涉及 React 组件的改动同时跑 `npm run typecheck`
- `onBrokenLinks` 配置为 `throw`，断链会让 build 直接失败，必须先修好
- 涉及首页或导航的改动还要本地 `npm run start` 目测一遍

## 提 commit / PR 时

- 仓库默认分支 `main`
- commit message 中文 / 英文皆可，但要写清"动了什么 + 为什么"
- 推送前先确认 build 通过
- 不要主动 `git push`，除非用户明确要求
- 涉及 GitHub Pages 部署相关字段（url / baseUrl 等）的改动，commit 前必须再次和用户确认
